# Security Audit

Date: 2026-04-06

## Scope

This audit covers the current Viact codebase and shipped examples:

1. `packages/framework`
2. `packages/adapter-node`
3. `packages/adapter-cloudflare`
4. `packages/vite-plugin`
5. `packages/cli`
6. `examples/basic`
7. `docs/*`
8. `.github/workflows/*`
9. Dependencies via `pnpm audit`

## Verdict

Viact has a sound high-level trust model in a few places, but the current implementation is not production-ready from a security standpoint.

Roughly:

1. Framework runtime security: `3/10`
2. Adapter and server boundary hardening: `4/10`
3. Example app security posture: `2/10`
4. Supply-chain and release posture: `6/10`
5. Overall today: `3/10`

The biggest issues are concrete, not theoretical: an inline-script XSS primitive, missing CSRF enforcement, and API routes bypassing middleware despite the docs claiming otherwise.

## What Is Good Today

1. Viact's server-owned loader model is directionally strong. Loader code stays server-side and is not bundled into the browser by default.
2. API routes use explicit method exports and return `405` when a handler is missing.
3. Route matching and manifest wiring are explicit and auditable.
4. `pnpm audit` reported no known vulnerable dependencies at the time of this audit.

## Findings

1. High: XSS via unescaped hydration state in an inline `<script>`.
   - References: `packages/framework/src/runtime.ts:404-423`
   - `buildHtmlDocument()` injects `window.__VIACT_STATE__=${JSON.stringify(hydrationState)}` directly into HTML.
   - If any loader returns attacker-controlled content containing `</script>`, that string can break out of the inline script block and execute arbitrary JavaScript in the page origin.
   - This is the most serious issue in the codebase because it turns normal server-rendered data into an XSS sink.

2. High: No CSRF enforcement for actions, despite docs claiming same-origin validation exists.
   - References: `packages/framework/src/runtime.ts:233-245`, `docs/DATA_LOADING.md:116-119`, `docs/ARCHITECTURE.md:244-250`
   - Non-safe methods are treated as actions and executed immediately.
   - There is no `Origin` check, no `Referer` check, and no CSRF token validation before `routeModule.action(routeArgs)` runs.
   - Any cookie-backed mutation endpoint is therefore CSRFable.
   - This is especially risky because the docs currently tell developers the framework protects them here.

3. High: API routes bypass middleware and any middleware-based auth entirely.
   - References: `packages/framework/src/runtime.ts:135-170`, `packages/framework/src/runtime.ts:187-216`, `docs/DATA_LOADING.md:230-235`
   - API route dispatch happens before the middleware chain and returns early.
   - The docs say API routes share the same context and middleware system, but the implementation does not do that.
   - A developer could reasonably believe auth, rate limiting, or logging middleware protects `/api/*` when it currently does not.

4. Medium: The example app accepts any `session=` cookie as authenticated.
   - References: `examples/basic/src/middleware/auth.ts:3-8`, `examples/basic/src/routes/dashboard.tsx:3-9`, `e2e/basic.test.ts:65-68`
   - Authentication is implemented as `request.headers.get("cookie")?.includes("session=")`.
   - There is no signature check, no expiry check, no binding to a server-side session store, and no secure cookie guidance in the example.
   - As example code, this is dangerous because users often copy the auth pattern from starter apps.

5. Medium: Host and protocol are derived from untrusted request headers.
   - References: `packages/adapter-node/src/index.ts:160-181`, `packages/vite-plugin/src/index.ts:284-317`, `packages/cli/bin/viact.js:252-257`
   - Request URLs are built from raw `Host` and `X-Forwarded-Proto` values.
   - In a misconfigured proxy chain, attacker-controlled forwarding headers can poison `args.url`, absolute redirects, canonical URLs, signed-link generation, or cookie policy logic built on top of Viact.
   - This is a classic host-header poisoning footgun.

6. Medium: Request bodies are fully buffered with no size limit.
   - References: `packages/adapter-node/src/index.ts:171-217`, `packages/vite-plugin/src/index.ts:305-314`
   - Both the Node adapter and dev SSR middleware read the entire request body into memory before routing.
   - There is no size cap, early rejection, or streaming path.
   - Large requests can cause memory pressure or denial of service.

7. Medium: `viact preview` has a path traversal risk in static file serving.
   - References: `packages/cli/bin/viact.js:179-236`
   - The preview server joins `clientDir` with raw `req.url` and serves the resulting file if it exists.
   - Because `join()` normalizes `..` segments, requests such as `/../../package.json` can escape `dist/client` and read files elsewhere under the project root if present.
   - This is limited to the preview server, but it is still a real file disclosure bug.

8. Medium: No default browser security headers are set on framework-generated HTML.
   - References: `packages/framework/src/runtime.ts:426-429`, `packages/adapter-node/src/index.ts:66-73`, `packages/cli/bin/viact.js:198-199`
   - HTML responses only set `Content-Type`.
   - There is no default CSP, no `X-Content-Type-Options`, no clickjacking protection, and no `Referrer-Policy`.
   - This makes exploitation easier when another bug exists, especially the hydration-state XSS issue.

9. Low: GitHub Actions use floating third-party action tags with write-capable permissions.
   - References: `.github/workflows/release.yml:11-45`, `.github/workflows/ci.yml:16-23`
   - Actions are pinned to version tags like `@v4` and `@v1`, not commit SHAs.
   - The release workflow has `contents: write`, `pull-requests: write`, and `id-token: write`.
   - This is not an active exploit in the repo, but it is weaker than current supply-chain hardening best practice.

## Docs Mismatches That Increase Risk

1. `docs/DATA_LOADING.md` says actions validate same-origin requests automatically, but the runtime does not implement that.
2. `docs/ARCHITECTURE.md` describes CSRF validation in the action lifecycle, but that check does not exist.
3. `docs/DATA_LOADING.md` says API routes share the middleware system, but API route dispatch currently bypasses middleware.

These mismatches matter because they create false security assumptions for framework users.

## Coverage Gaps

1. There are no tests covering XSS-safe serialization of hydration state.
2. There are no tests covering CSRF rejection for action requests.
3. There are no tests proving API routes inherit middleware or auth.
4. There are no tests for static-file traversal in `viact preview`.
5. There are no tests asserting default security headers.

## Priority Fix Order

1. Fix hydration-state serialization so inline state cannot terminate the script block.
2. Implement real CSRF protection for actions before continuing to document cookie-backed auth flows.
3. Run middleware for API routes or explicitly add API-specific middleware configuration and document it accurately.
4. Fix preview static-file path normalization so requests cannot escape `dist/client`.
5. Add request-body limits in the Node adapter and dev SSR middleware.
6. Add a default security header baseline or a first-class way to configure one safely.
7. Replace the example auth check with a signed-session example or clearly label it as intentionally fake demo auth.
8. Pin GitHub Actions to immutable SHAs.

## Bottom Line

Viact's architecture is moving in a reasonable direction, but the current implementation still has multiple high-severity web security issues. I would not treat the framework or the example app as safe for internet-facing production use until the XSS, CSRF, and API-middleware gaps are fixed and covered by tests.
